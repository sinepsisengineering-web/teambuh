## Фиксированная структура (шпаргалка)
### Корень проекта на сервере
- `/srv/teambuh`

### Ключевые каталоги
- `/srv/teambuh/master` — Traefik + master-api
- `/srv/teambuh/template` — эталон для новых тенантов
- `/srv/teambuh/tenants/<tenant_id>` — отдельный тенант
- `/srv/teambuh/backups` — архивы бэкапов

### Ключевые файлы для текущего сценария
- `/srv/teambuh/server/index.js` — основной API (в т.ч. rules)
- `/srv/teambuh/scripts/init-fresh-instance.js` — инициализация БД
- `/srv/teambuh/scripts/prepare-template.sh` — пересборка template
- `/srv/teambuh/scripts/update-tenants.sh` — обновление существующих тенантов
- `/srv/teambuh/template/.dockerignore` — не должен исключать `dist`
- `/srv/teambuh/template/Dockerfile` — Dockerfile для тенанта (runtime, без сборки из исходников)

### Базы правил (что где хранится)
- Системные правила (общие):  
`/srv/teambuh/tenants/<tenant_id>/data/global_data/rules.db`
- Правила конкретной фирмы (tenant):  
`/srv/teambuh/tenants/<tenant_id>/data/client_data/org_default/db/rules.db`
- Эталонная БД для новых клиентов:  
`/srv/teambuh/template/data/global_data/rules.db` (серверный источник, не перезаписывается из Git)

Ожидаемое состояние после выравнивания:
- `global_data/rules.db` = 32 системных правила
- `client_data/.../rules.db` после sync = 32 (плюс custom при наличии)

## Фиксированный порядок действий (без подбора)
1. Бэкап (`master`, `template`, `tenants`).
2. Обновление кода на сервере через GitHub (`git pull origin main`).
3. Пересборка template (без перезаписи серверной rules.db):  
`cd /srv/teambuh && ./scripts/prepare-template.sh`
4. Обновление нужного тенанта:  
`cd /srv/teambuh && sudo ./scripts/update-tenants.sh <tenant_id>`
или массово:
`cd /srv/teambuh && sudo ./scripts/update-tenants.sh --all`
5. Проверка сайта:  
`curl -I https://<tenant_id>.teambuh.ru`
6. Проверка правил:
- `sqlite3 .../global_data/rules.db "select count(*) from task_rules;"`
- `sqlite3 .../client_data/org_default/db/rules.db "select count(*) from task_rules;"`
7. Если global старая (ошибка `no column named version`), для конкретного тенанта:
- бэкап старой global `rules.db`;
- удалить `rules.db` + `-shm/-wal`;
- `docker compose restart` в папке тенанта;
- повторная синхронизация правил.

## Что не делать
- Не редактировать tenant `data/` без бэкапа.
- Не запускать массовые обновления без проверки на одном тестовом тенанте.
- Не проверять права доступа только по UI: обязательно подтверждать API/БД.

## Команды обновления (фиксировано)
- Единая команда (рекомендуется):
`cd /srv/teambuh && sudo bash ./scripts/deploy-tenants.sh test-2`
- Единая команда для всех:
`cd /srv/teambuh && sudo bash ./scripts/deploy-tenants.sh --all`
- Единая команда preview без сборки:
`cd /srv/teambuh && sudo bash ./scripts/deploy-tenants.sh --dry-run --all`

Старый раздельный режим (если нужно):
- Обновить одного клиента:
`cd /srv/teambuh && sudo ./scripts/update-tenants.sh test-2`
- Обновить всех клиентов:
`cd /srv/teambuh && sudo ./scripts/update-tenants.sh --all`
- Предпросмотр без сборки (проверить что будет обновлено):
`cd /srv/teambuh && sudo ./scripts/update-tenants.sh --dry-run --all`

Примечание:
- Скрипт сам проверяет, что `template` готов (`Dockerfile` и `dist/index.html`).
- Скрипт сам убирает строку `dist` из tenant `.dockerignore`, чтобы не ломалась сборка.

- Новые тенанты создаются автоматически и сразу рабочие (DNS+SSL+frontend+backend).

- Существующие тенанты обновлены без потери клиентских данных.
