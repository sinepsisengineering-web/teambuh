# TeamBuh — План работ только по серверу

## Цель
Привести серверную часть к стабильному состоянию: убрать наследие старой схемы правил, зафиксировать права доступа, обновить шаблон тенанта, обновить существующие тенанты и подтвердить, что новые регистрации создают полностью рабочие инстансы.

## Границы плана
- Только сервер: Docker, Traefik, DNS, `master`, `scripts`, `template`, `tenants`, SQLite.
- Без фронтенд-дизайна и без новых UI-страниц (кроме подготовки API-основы под будущую страницу супер-админа).

---

## Этап 0. Бэкап и точка восстановления
1. Сделать бэкап:
- `/srv/teambuh/master/tenants.db`
- `/srv/teambuh/master/traefik/dynamic/`
- `/srv/teambuh/template/`
- `/srv/teambuh/tenants/`
2. Проверить, что бэкап открывается и читается.
3. Зафиксировать текущие версии контейнеров и `git rev-parse HEAD`.

Критерий готовности:
- Есть полный бэкап, подтверждено чтение архива.

---

## Этап 1. Устранить смешение старой и новой модели правил
1. Зафиксировать единственный источник системных правил: `server/database/rulesMigration.js`.
2. Убрать конфликт с устаревшей инициализацией системных правил из `scripts/init-fresh-instance.js`.
3. Пересоздать `template/data/global_data/rules.db` только в новой схеме.
4. Проверить структуру `task_rules` (наличие новых полей: `version`, `task_type`, `short_title`, и т.д.).

Критерий готовности:
- В `global_data/rules.db` только новая схема и полный набор системных правил (ожидаемо 32).

---

## Этап 2. Закрыть права на системные (налоговые) правила в тенантах
1. Удалить дублирующиеся routes правил в `server/index.js`.
2. Оставить только один набор API-эндпоинтов правил с серверной проверкой:
- `source=system` нельзя `POST/PUT/DELETE` на tenant-уровне.
3. Проверить, что директор (`admin`) может работать только с `custom` правилами.
4. Проверить, что глобальные системные правила доступны только через отдельный super-admin контур.

Критерий готовности:
- Любая попытка изменить системное правило в тенанте дает `403`.

---

## Этап 3. Исправить шаблон создания новых тенантов
1. Проверить `template/.dockerignore`: `dist` не должен исключаться.
2. Проверить, что `template` содержит:
- корректный `dist/`
- актуальный `server/`
- актуальные `scripts/`
- корректный `data/global_data/rules.db` новой схемы.
3. Проверить `master/docker-compose.yml`:
- volume `./traefik/dynamic:/srv/master/traefik/dynamic` для `master-api` присутствует.

Критерий готовности:
- Новый тенант создается с рабочим фронтом и корректной серверной структурой.

---

## Этап 4. Обновление уже существующих тенантов
1. Подготовить безопасный update-скрипт с dry-run.
2. Для каждого тенанта:
- обновить код (`server`, `scripts`, `dist`),
- не трогать клиентские данные (`data/client_data/*`),
- выровнять `data/global_data/rules.db` на новую схему,
- выполнить синхронизацию правил tenant DB.
3. Собрать отчет по каждому тенанту: до/после (count системных/tenant rules, статус контейнера).

Критерий готовности:
- По всем тенантам синхронизация правил проходит без ошибок schema/constraint.

---

## Этап 5. DNS/SSL и маршрутизация
1. Подтвердить wildcard DNS:
- `*.teambuh.ru -> <IP сервера>`.
2. Проверить доступность новых поддоменов через публичный резолвер.
3. Проверить Traefik ACME (Let’s Encrypt) на новом тенанте.
4. Убедиться, что для нового поддомена нет `NXDOMAIN` и нет fallback на self-signed.

Критерий готовности:
- Новый поддомен открывается по HTTPS с валидным сертификатом.

---

## Этап 6. Контрольный E2E-тест серверного потока
1. Создать тестовую фирму через `https://teambuh.ru`.
2. Проверить:
- создана папка тенанта,
- контейнер запущен,
- Traefik роут создан,
- поддомен открывается,
- логин директора работает,
- системные правила видны, но не редактируются директором,
- custom-правила директором создаются/редактируются.
3. Удалить тестовый тенант корректным скриптом.

Критерий готовности:
- Полный SaaS-поток проходит без ручных правок после регистрации.

---

## Этап 7. Подготовка к будущей странице управления системными правилами
1. Зафиксировать серверный контракт (API) для super-admin:
- просмотр системных правил,
- редактирование системных правил,
- версия/публикация,
- механизм доставки обновлений в тенанты.
2. Ограничить API по роли `super-admin`.
3. Протокол обновления тенантов:
- «обновить только правила» без затрагивания клиентских данных.

Критерий готовности:
- Готов backend-контур для будущей отдельной страницы управления системными правилами.

---

## Риски и контроль
- Риск потери данных при миграции: решается обязательным бэкапом и dry-run.
- Риск регрессий из-за дублей роутов: решается удалением дублей и проверкой `403` на системные правила.
- Риск частичного обновления тенантов: решается отчетом по каждому тенанту и повторяемым скриптом.

---

## Definition of Done
- Новые тенанты создаются автоматически и сразу рабочие (DNS+SSL+frontend+backend).
- Системные налоговые правила единообразны, в новой схеме и не редактируются директором.
- Существующие тенанты обновлены без потери клиентских данных.
- Есть серверный регламент дальнейших обновлений.
